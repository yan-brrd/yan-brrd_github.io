SIEM Threat Detection
& Dashboard Build

Standing up a working SIEM environment and turning raw security logs into actionable monitoring — using the BOTSv2 dataset to simulate a real-world breach investigation.

Category · Detection & Monitoring Tools · Splunk Enterprise, BOTSv2, SPL Kill Chain · Reconnaissance → Exfiltration
// 01
Overview

This case covers standing up a working SIEM environment and learning to turn raw security logs into actionable monitoring. The lab uses BOTSv2 (Boss of the SOC v2) — a publicly available dataset that simulates real-world security incidents across multiple data sources.

The goal was to ingest heterogeneous logs, identify patterns of compromise, and build an intuitive dashboard that communicates findings to both technical and non-technical stakeholders.

Methodology
Step 01

Ingest & Orient

Loaded the BOTSv2 dataset into Splunk and ran broad searches to understand what sourcetypes and fields were available.

Step 02

Hunt with SPL

Wrote targeted searches to surface indicators of compromise across the simulated environment.

Step 03

Build Dashboards

Translated the most useful searches into permanent dashboard panels with time charts, tables, and single-value indicators.

Step 04

Rehearse & Present

Practiced presenting the dashboard live, narrating each panel's purpose and relevance to non-security audiences.

// 04
Key SPL Queries

Event Volume by Sourcetype

Counts events by sourcetype to see overall log volume across the dataset, identifying the most prolific data sources.

SPL
index=botsv2 sourcetype=* | stats count by sourcetype | sort -count

Event Volume Trends Over Time

Visualizes how event volume changes over time, broken down by sourcetype — revealing patterns and spikes that correspond to simulated attack phases.

SPL
index=botsv2 | timechart span=1h count by sourcetype
// 05
Key Findings
High-volume database activity: mysql:server:stats and mysql:transaction:details were the highest-volume sourcetypes (~13.5M+ events each), indicating heavy database activity logging and a potential exfiltration target.
Bursty event distribution: The hourly timechart revealed that log volume was extremely concentrated rather than continuous — most windows showed zero events, with two sharp spikes likely corresponding to simulated attack phases.
Multi-source correlation potential: The presence of HTTP, database, and system logs created opportunities for correlated detection of lateral movement and data exfiltration attempts across multiple data sources.

// 08
Reflection

The hardest part of this case was getting comfortable with SPL syntax — understanding how commands like stats, timechart, and sort pipe into each other to transform raw events into meaningful insights. Once that mental model clicked, the rest flowed naturally.

Building the dashboard taught me that technical prowess alone doesn't matter if you can't communicate findings clearly. Choosing the right visualization for each metric (sparklines for trends, single values for alerts, tables for detail) was as important as the underlying queries. Practicing the presentation made all the difference — I learned to tell the story of the data, not just show it.
